operator new confusion

nothrow is implemented by 

try
{
    operator new(sz)
}
catch(std::bad_alloc const&)
{
    return nullptr;
}


1. It still introduces the runtime of C++ exceptions which bloats your binary size for 100 kb. bad_alloc is still there internally.

2. nothrow const& is terrible since you pass a reference to a null object. That is a total waste of register. You pass an address to an empty object which is absurd.

3. std::bad_alloc is terrible since you have no way to handle allocation failure safely. Many OS has the overcommit mechanism. -- Cqwrteur

Absolute non-sense. Just implement void* operator new(std::size_t count, const std::nothrow_t&) if you don't want to throw, this wiki is for generally applicable standard conforming code --Ybab321 (talk) 03:29, 12 February 2021 (PST)

The clause in qualified lookup does not need restore.

Hello Ybab321:

"including those introduced into the global namespace by a using declaration" is missleading there, normal lookup rule for namespace member applies for the global namespace, but it implies that there's some difference for `::`, which is not true. Readers should just refer to the linked section below. It's not necessary to only mention using declarations, as if using directives were not found. If we want emphasis it here, the whole rule should be memtioned exhaustively, such as recursively nested inline namespaces and ones introduced by using in them, ...

It was introduced because the original expression is "declarations **made** in the global namespace", thus using declarations do make a difference, but I changed it to declarations in the global namespace, which means the complier behaves as other lookups for declarations, where they are **made** is not taken into account.

OK that's fair, I can see how that comment would mislead readers into thinking there's a special rule for global namespace lookup. Thanks for the rationale, it's much appreciated --Ybab321 (talk) 03:54, 9 October 2024 (PDT)
